## 3v@l
[3v@l](https://play.picoctf.org/practice/challenge/484?category=1&page=1&search=3v%40l)

This seems to be a command injection challenge. Looking at `Inspect`, there is a piece of comment that describes how the server is sanitizing this input for `eval()`. 

I note the blacklisted commands, but also this regex:

    r'0x[0-9A-Fa-f]+|\\u[0-9A-Fa-f]{4}|%[0-9A-Fa-f]{2}|\.[A-Za-z0-9]{1,3}\b|[\\\/]|\.\.'

Breaking this down:
    0x[0-9A-Fa-f]+ This matches any hexadecimal numbers, like 0xff
    \\u[0-9A-Fa-f]{4} This matches strings like \u4123, where we have 4 hex values concretely
    %[0-9A-Fa-f]{2} This matches strings like %2a, probably URL-encoded
    .[A-Za-z0-9]{1,3}\b This may match strings like .abc, .023, .2 (matching 1 to 3 characters)
    [\\\/] This looks odd, but "\" is used for literal characters so here, it matches "\" or "/"
    \.\. This matches two dots ..

In addition, looking up on `eval()`, it supports Python code, not Bash code.

This may require some additional time. I definitely look up some resources regarding `eval() code injection` regarding this problem.

There is this default piece of Python code.

    __import__('os').system(command)

The `command` part is where we put our command. Note that the server sanitizes 'os' so we can split 'os' into 'o' + 's'. If the command is on the list of sanitized commands, then we could use a similar strategy.

We also take note of the regex for this to work.
